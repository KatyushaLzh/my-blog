# 信号量与同步

第 16 讲的核心不是再引入一个新的锁 API，而是把“同步”从互斥问题中单独抽出来：同步关心的是某个事件、条件或资源状态必须先发生，另一个线程才能继续执行。

互斥解决的是“同一时刻谁能访问共享状态”；同步解决的是“哪个时刻之后程序才允许继续”。信号量就是一种用计数器表达同步条件的经典机制。

## 同步点与 Happens-Before 关系

同步点可以理解为程序达到某个全局确定状态的瞬间。等待同步的线程并不是单纯为了避免数据竞争，而是在等待一个条件成为真：

```c
while (!sync_condition()) {
    wait();
}
proceed();
```

例如：

- 所有子线程都已经结束；
- 缓冲区中至少有一个可消费对象；
- 计算图中某个节点的所有前驱都已经完成；
- 某个生产者已经产生出消费者所需的资源。

一旦同步条件成立，程序中就形成了一对顺序约束：

```text
条件达成前的操作 happens-before 条件达成后的操作
```

因此，同步原语的本质是把不确定的线程交错重新约束成可证明的先后关系。

## 互斥锁同步模型的局限性

讲义中先给出了一个“用互斥锁实现同步”的思维实验。对计算图中的一条依赖边 `u -> v`，分配一把锁 `e`：

```c
main: lock(e); spawn_threads();
u:    work(u); unlock(e);
v:    lock(e); work(v);
```

`main` 先把锁拿走，使 `v` 无法继续；`u` 完成后释放这把锁，`v` 才能获得锁并开始执行。这样就用 `unlock(e) -> lock(e)` 的 release-acquire 语义实现了：

```text
work(u) happens-before work(v)
```

这个例子说明，互斥锁的 `release` 和 `acquire` 天然带有内存顺序约束，能够表达一次临时的同步事件。

但这只是理解模型，不是规范的 pthread 写法。`pthread_mutex_t` 通常要求加锁线程和解锁线程一致；让 `main` 加锁、`u` 解锁属于 undefined behavior。它的价值在于暴露出更一般的抽象：如果一个同步凭证可以被某个线程释放、被另一个线程获取，那么它就能表达跨线程的先后关系。

## 信号量的基本抽象

信号量可以看作一个带阻塞等待能力的计数器：

```text
semaphore = count + wait queue
```

其中 `count` 表示当前可被消费的凭证数量。信号量提供两个基本操作：

- `P` / `wait` / `down`：申请一个凭证；
- `V` / `post` / `up` / `signal`：释放一个凭证。

它们的语义是：

```c
P(sem):
    wait until sem->count > 0;
    sem->count--;

V(sem):
    sem->count++;
    wake up possible waiters;
```

`P` 成功返回时，程序可以保证曾经存在一个可消费的凭证；`V` 则创建一个新的凭证，并可能唤醒等待者。初始值 `count = k` 可以理解为系统一开始已经执行了 `k` 次 `V`。

## 信号量的实现结构

讲义中的教学实现可以写成 `mutex + condition variable`：

```c
void P(sem_t *sem) {
    mutex_lock(&sem->lk);
    while (!(sem->count > 0)) {
        cond_wait(&sem->cv, &sem->lk);
    }
    sem->count--;
    mutex_unlock(&sem->lk);
}

void V(sem_t *sem) {
    mutex_lock(&sem->lk);
    sem->count++;
    cond_broadcast(&sem->cv);
    mutex_unlock(&sem->lk);
}
```

这里 `mutex` 保护 `count` 的检查和修改，`condition variable` 负责在 `count == 0` 时阻塞等待。这个版本适合解释语义，但不是唯一实现方式。

真实系统中的信号量通常会分成快路径和慢路径：

- 快路径：使用原子操作检查并修改计数器；
- 慢路径：当计数不足时，进入等待队列、futex 或内核调度机制；
- 唤醒路径：`V` 修改计数后唤醒可能阻塞的线程。

因此，信号量的底层本质不是“单靠原子操作”，也不是“必须由条件变量实现”，而是“原子计数 + 阻塞唤醒机制”。条件变量只是教学上方便的一种上层模拟方式。

## 信号量与互斥锁的关系

当信号量初始值为 `1` 时，它可以实现互斥锁：

```c
sem_t sem = SEM_INIT(1);

void lock() {
    P(&sem);
}

void unlock() {
    V(&sem);
}
```

此时系统中最多只有一个凭证，所以最多只有一个线程能通过 `P` 进入临界区。从这个意义上说，互斥锁可以看作二值信号量的特例。

但二者的语义重点不同：

- 互斥锁强调所有权和临界区保护，通常要求同一线程加锁和解锁；
- 信号量强调凭证数量和同步事件，`V` 与 `P` 可以发生在不同线程中；
- 互斥锁保护“谁能访问共享状态”，信号量表达“还有多少个许可可以被消费”。

因此，在写程序时不要把信号量简单当成“更通用的锁”。它更适合表达计数型资源和事件配对。

## 一次性事件同步

信号量可以直接表达一次临时的 happens-before 关系：

```c
sem_t done = SEM_INIT(0);

void A() {
    work_A();
    V(&done);
}

void B() {
    P(&done);
    work_B();
}
```

由于 `done` 初始为 `0`，`B` 在 `P(&done)` 处必须等待。只有 `A` 执行 `V(&done)` 后，`B` 才可能继续。因此可以得到：

```text
work_A() happens-before work_B()
```

这正是前面“用互斥锁实现同步”思维实验的规范版本。互斥锁 hack 中的“放回钥匙”被替换成了信号量的 `V`，等待者通过 `P` 消费这个事件。

## 并发数量控制

信号量的另一个典型用途是限制并发数量。若某类资源最多允许 `n` 个线程同时使用，可以令：

```c
sem_t quota = SEM_INIT(n);

void worker() {
    P(&quota);
    use_limited_resource();
    V(&quota);
}
```

`quota` 的计数器表示当前剩余的资源许可。只要每个线程进入前执行 `P`、离开后执行 `V`，系统中同时使用该资源的线程数就不会超过 `n`。

这个模型适用于停车场空位、游泳馆手环、餐厅桌位、连接池容量、线程池任务配额等具有自然数量含义的问题。

## 线程 Join 与计算图同步

信号量可以实现线程 `join`。若主线程需要等待 `n` 个工作线程结束，可以使用一个初始值为 `0` 的信号量：

```c
sem_t done = SEM_INIT(0);

void worker() {
    work();
    V(&done);
}

void main_thread() {
    for (int i = 0; i < n; i++) {
        P(&done);
    }
}
```

每个工作线程结束时释放一个凭证，主线程需要消费 `n` 个凭证才能继续。这里 `done.count` 表示“已经完成但尚未被主线程收集的线程数量”。

在一般计算图中，边 `u -> v` 表示 `v` 必须等待 `u` 完成。可以为每条边分配一个初始值为 `0` 的信号量：

```c
u: work(u); V(edge_u_v);
v: P(edge_u_v); work(v);
```

也可以为每个节点维护一个计数型信号量，使节点在收齐所有前驱凭证后再执行。信号量适合这种模型，是因为依赖关系可以被转化为“需要消费多少个完成事件”。

## 生产者消费者模型

生产者消费者是信号量最自然的应用之一。对于容量为 `depth` 的缓冲区，可以定义：

```c
sem_t empty = SEM_INIT(depth);
sem_t fill  = SEM_INIT(0);
```

其中：

- `empty` 表示当前空槽数量；
- `fill` 表示当前可消费对象数量。

基本结构是：

```c
void produce() {
    P(&empty);
    put_item();
    V(&fill);
}

void consume() {
    P(&fill);
    get_item();
    V(&empty);
}
```

生产者先消费一个空槽，再产生一个满槽；消费者先消费一个满槽，再释放一个空槽。系统中的核心不变量是：

```text
empty + fill + producer_holding + consumer_holding = depth
```

如果 `put_item()` 和 `get_item()` 操作真实共享队列，还需要额外的互斥锁保护队列结构。此时信号量负责资源数量，互斥锁负责数据结构一致性。两者职责不同，不能互相替代。

## 与条件变量的表达能力对比

信号量等待的是一个计数器：

```text
count > 0
```

条件变量等待的是共享状态上的任意谓词：

```text
predicate(shared_state) == true
```

因此，信号量在同步条件可以自然转化为数量时非常简洁，例如：

- 缓冲区中有多少个元素；
- 还剩多少个空槽；
- 已经完成多少个子任务；
- 还允许多少个线程进入某个区域。

但当同步条件是复杂的全局状态时，单个计数器往往难以表达。例如：

```c
avail[lhs] && avail[rhs]
can_put && !pending_close
pred_done == n_pred
```

这些条件更适合用 `mutex + condition variable` 表达：

```c
mutex_lock(&lk);
while (!predicate(shared_state)) {
    cond_wait(&cv, &lk);
}
update_shared_state();
mutex_unlock(&lk);
```

所以二者的关键区别可以概括为：

- 信号量等待一个 token；
- 条件变量等待一个 predicate。

理论上，条件变量可以用信号量和额外状态模拟；反过来，信号量也可以用互斥锁和条件变量实现。但工程上不应只看可实现性，还要看同步条件是否能被清晰建模。

## 信号量建模的适用边界

信号量不是万能同步方法。讲义中的哲学家吃饭问题说明了这一点。

每个哲学家需要同时获得左右两把叉子，对应条件是：

```c
avail[lhs] && avail[rhs]
```

若直接写成：

```c
P(&fork[lhs]);
P(&fork[rhs]);
```

可能出现所有哲学家都拿起左手叉子、再一起等待右手叉子的循环等待，进而发生死锁。问题不在于 `P/V` 本身错误，而在于“同时获得两份资源”这个全局条件没有被一个简单计数器完整表达。

可以通过额外协议修复，例如：

- 限制最多只有四个哲学家同时尝试进餐；
- 对叉子编号，所有线程按固定顺序获取资源；
- 引入调度者，由调度者根据全局状态决定谁可以继续。

这些方案都说明：使用信号量时必须明确计数器到底代表什么不变量。若计数器无法自然对应同步条件，代码会变得复杂，正确性证明也会变难。

## 条件变量模拟的原子性问题

用信号量模拟条件变量时，最困难的问题是避免漏唤醒。一个朴素实现可能写成：

```c
void wait(cond_t *cv, mutex_t *mutex) {
    cv->nwait++;
    mutex_unlock(mutex);
    P(&cv->sleep);
    mutex_lock(mutex);
}

void broadcast(cond_t *cv) {
    mutex_lock(&cv->lock);
    for (int i = 0; i < cv->nwait; i++) {
        V(&cv->sleep);
    }
    cv->nwait = 0;
    mutex_unlock(&cv->lock);
}
```

这个模型的危险在于：线程可能已经计入等待者数量，但还没有真正执行到 `P(&cv->sleep)`；此时另一个线程执行 `broadcast`，释放的凭证可能被错误线程消费，或导致后续等待者错过唤醒。

条件变量的 `cond_wait(&cv, &mutex)` 必须保证“释放互斥锁”和“进入等待状态”之间没有可见空窗。若这个 release-wait 动作不能做成不可分割的原子操作，就会出现 lost wakeup。真实系统通常需要 futex、等待队列或内核调度机制来完成这一步。

## 同步原语的粒度差异

信号量和条件变量的粒度差异不是“谁的锁更细”，而是建模对象不同。

信号量的粒度是 token 或资源单位。它把资源数量拆成可消费的许可，适合把同步条件写成 `count > 0` 的问题。生产者消费者中，`empty` 和 `fill` 分别对应空槽和满槽，粒度非常清楚。

条件变量的粒度是共享状态交接点。它通常配合一把互斥锁保护一组共享不变量，等待者在 `while (!predicate)` 中睡眠，修改者更新状态后唤醒等待者。等待期间线程不持有互斥锁，因此锁只覆盖条件检查和状态修改，不覆盖睡眠时间。

因此：

- 计数型资源问题中，信号量通常更直接；
- 复杂谓词同步中，条件变量通常更清晰；
- 二者都需要控制临界区范围，把主要计算放到同步原语之外。

## 使用原则

使用信号量时，首先要回答“计数器代表什么”：

- 若它代表剩余资源数，`P` 是占用资源，`V` 是释放资源；
- 若它代表已发生事件数，`V` 是发布事件，`P` 是消费事件；
- 若它代表并发配额，`P` 是进入受限区域，`V` 是离开受限区域。

一个信号量对应一个清晰的不变量。若无法写出这个不变量，通常说明问题更适合使用条件变量、互斥锁保护的共享状态，或者额外的调度协议。

实践中可以按以下顺序选择同步原语：

- 只需要保护共享数据结构：优先使用互斥锁；
- 需要等待任意共享状态条件：优先使用条件变量；
- 需要表达资源数量、事件次数或并发配额：优先使用信号量；
- 条件复杂且涉及多个资源：先写出全局不变量，再决定是否需要额外调度者或锁顺序。

## 小结

信号量是一个带阻塞等待能力的计数器。它通过 `P` 消费凭证、通过 `V` 释放凭证，从而表达资源数量、事件配对和 happens-before 关系。

第 16 讲的主线可以概括为：

1. 同步的本质是建立跨线程的先后关系。
2. 互斥锁的 release-acquire 语义可以帮助理解同步，但跨线程解锁 mutex 不是规范实现。
3. 信号量把“一个凭证”推广成“多个凭证”，从而支持事件同步和资源计数。
4. 生产者消费者是信号量最自然的模型，因为同步条件本身就是数量。
5. 条件变量更适合任意谓词同步，信号量更适合计数型同步。
6. 所有同步程序最终都要回到共享状态、不变量和 happens-before 关系的证明。
